# 0002 - Work Item Hierarchy (Epic / Story / Task)

- Status: accepted
- Date: 2026-03-06

## 배경
현재 저장소는 목표/규칙/디렉터리 구조는 정리되어 있으나, 실제 실행 단위를 일관되게 정의하는 기준이 없다.  
그 결과 실험(`experiments`), 공용화(`tools`), 데이터 산출(`data/artifacts`, `data/final`) 작업을 계획할 때
- 작업 크기 편차가 커지고
- 완료 판단 기준이 모호해지며
- 협업 시 인수인계/추적성이 떨어지는 문제가 발생할 수 있다.

따라서 팀 공통의 업무 단위 체계를 명시적으로 채택할 필요가 있다.

## 결정
업무 단위를 **에픽(Epic) → 스토리(Story) → 태스크(Task)** 3계층으로 표준화한다.

### 1) 에픽(Epic)
큰 사용자/프로젝트 가치 단위로 정의한다.
- 표준 문장: `[큰 목표] 달성을 위해 [가설]을 검증한다.`
- 필수 항목: 목표, 가설, 범위(포함/제외), 완료 조건(Verify 기준), Release 산출물
- 권장 규모: 다수 스토리를 포함하는 중기 단위

### 2) 스토리(Story)
검증 가능한 사용자 가치 단위로 정의한다.
- 표준 문장: `[사용자]로서, [목적]을 위해 [기능]을 할 수 있다.`
- 필수 항목: 수용 기준(권장: Given/When/Then)
- 완료 판정: 수용 기준 충족 여부로만 판단

### 3) 태스크(Task)
스토리 구현을 위한 실행/기술 단위로 정의한다.
- 표준 문장: `[대상]을 [작업]한다.`
- 필수 항목: 작업 내용(동사 중심), 산출물 경로(파일/디렉터리), 완료 조건(검증 명령/검토 방법)
- 권장 규모: 0.5~1일 내 완료 가능한 단위


## 제목 규칙
- 에픽/스토리/태스크의 **이슈 제목은 한줄 카드와 동일**하게 작성한다.
- 권장 접두사: `[Epic]`, `[Story]`, `[Task]`

## 운영 규칙
1. 모든 스토리는 정확히 1개의 에픽에 소속된다.
2. 모든 태스크는 정확히 1개의 스토리에 소속된다.
3. 스토리는 수용 기준 없이는 생성/완료 처리할 수 없다.
4. 태스크는 산출물 경로와 검증 방법 없이는 완료 처리할 수 없다.
5. 결과 산출물은 저장소 규칙을 따른다.
   - 실험성 산출물: `experiments/epicNNN_*/outputs`
   - 재사용 도구: `tools/`
   - 중간 산출물: `data/artifacts/`
   - 최종 데이터: `data/final/`

## 결과
### 장점
- 계획/실행/검증의 기준이 통일되어 협업 효율이 향상된다.
- 작업 추적성(요구사항→실행→산출물)이 좋아진다.
- 완료 기준이 명확해져 일정 예측 가능성이 높아진다.

### 단점/비용
- 초기에 에픽/스토리/태스크 문서화 오버헤드가 증가한다.
- 작은 변경에도 수용 기준/DoD 정리가 필요해 관리 비용이 발생한다.

## 도입 계획
1. 다음 스프린트부터 신규 작업은 본 규칙을 적용한다.
2. 기존 진행 중 작업은 영향이 큰 항목부터 점진적으로 전환한다.
3. `experiments/epicNNN_*/README.md`에 스토리/태스크 링크 또는 식별자를 기록한다.

## 짧은 템플릿
### 에픽(Epic)
- 형식: `[큰 목표] 달성을 위해 [가설]을 검증한다.`
- 필수 항목: `목표 / 가설 / 범위(포함/제외) / 완료 조건(Verify 기준) / Release 산출물`

### 스토리(Story)
- 형식: `[사용자]로서, [목적]을 위해 [기능]을 할 수 있다.`
- 필수 항목: `수용 기준 1~3개`

### 태스크(Task)
- 형식: `[대상]을/를 [작업]한다.`
- 필수 항목: `산출물 경로 / 검증 방법`
